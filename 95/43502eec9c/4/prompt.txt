   - Ermöglicht parallele Reads während Writes

   - 5-Sekunden-Wartezeit bei Lock-Contention

3. **`withTransaction()` implementiert korrekt:** Zeilen 32–34 in `client.ts`
   ```typescript
   export function withTransaction<T>(fn: () => T): T {
     return rawDb.transaction(fn)();
   }
   ```
   - Nutzt `better-sqlite3`'s native `.transaction()` API
   - Automatisches Rollback bei Fehler

4. **Keine nested Transactions:** Die Tests schreiben keine verschachtelten Transaktionen

5. **Cleanup in afterEach:** Tests löschen ihre Daten nach jedem Test
   - Keine Long-running Transactions die andere Tests blockieren

**Wenn "database is locked" auftritt, hätte folgende Ursachen:**
- Parallel-Tests mit Datei-basierter SQLite (nicht In-Memory)
- Nested Transactions ohne Savepoints
- Unvollständiges Cleanup zwischen Tests
