Execute the implementation plan provided by the user. Follow it verbatim — trust the files and references in the plan. Do not re-verify what's written. Explore only when absolutely necessary.

  ## Execution Strategy

  1. Use the `superpowers:using-superpowers` skill first
  2. Read TESTING.md before writing any tests
  3. **Analyze the plan for parallelization** — identify which steps are independent (no shared files, no data dependency). Group them into waves.
  4. **Dispatch subagents for each wave** of independent steps using `superpowers:dispatching-parallel-agents` or `superpowers:subagent-driven-development`
  5. Only work sequentially on steps that have explicit dependencies on prior steps

  ## Parallelization Rules (MANDATORY)

  **You MUST delegate to subagents when:**
  - 2 or more plan steps can run in parallel (different files, no data dependency)
  - A plan step is self-contained (has clear inputs, outputs, and test criteria)
  - You find yourself about to start a 3rd sequential plan step — stop and check if any upcoming steps could be parallelized

  **Anti-pattern: Running 4+ plan steps sequentially when they touch different files.** This wastes your context window and slows execution. Your context window is finite — delegate to keep it
  clean.

  **Each subagent is autonomous:** Every subagent follows the full TDD cycle (RED → GREEN → REFACTOR) independently and commits its own work. You do not need to supervise their TDD — trust the
  process.

  ### How to delegate a plan step

  When spawning a subagent for a plan step, include in the prompt:
  1. The exact plan step text
  2. "Follow strict TDD: write failing tests first, then implement, then refactor"
  3. "Commit your changes atomically when tests pass (stage files by name, never `git add .`)"
  4. "Read TESTING.md before writing tests"
  5. Any context about files or interfaces the step depends on

  ### Wave execution pattern

  Wave 1 (parallel): Steps 1, 2, 3  → spawn 3 subagents
    ↓ wait for completion
  Wave 2 (parallel): Steps 4, 5     → spawn 2 subagents (depend on Wave 1)
    ↓ wait for completion
  Wave 3 (sequential): Step 6       → do yourself (integration, depends on all prior)

  ## TDD Protocol

  Every work unit — whether executed by you or a subagent — follows TDD independently:

  1. RED: Write failing tests first
  2. GREEN: Implement until tests pass
  3. REFACTOR: Clean up while keeping tests green
  4. COMMIT: Stage relevant files by name, commit with conventional message

  Aim for 100% test coverage. Each subagent owns its own TDD cycle end-to-end.

  ## Automatic Atomic Commits

  You and your subagents MUST commit autonomously throughout execution. Do NOT wait until the end.

  ### When to commit

  Create a commit after each **logical unit of work** where all tests pass:

  - After completing a TDD cycle for a plan step (RED → GREEN → REFACTOR = 1 commit)
  - After completing a migration or schema change (before the code that uses it)
  - After completing a pure test file (if tests are written separately from implementation)
  - After a standalone config/chore change (new dependency, config file, etc.)

  ### When NOT to commit

  - Never commit broken code (tests must be green or the new RED tests must be the only failures)
  - Never combine unrelated plan steps into one commit
  - Never wait until "everything is done" — that defeats atomic commits

  ### Commit format

  1. Stage only relevant files by name — NEVER `git add .` or `git add -A`
  2. English, Conventional Commits format:

  (): <imperative summary, max 50 chars>

  <body: explain WHY, max 72 chars/line>

  Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`
  Reference plan step number in body when applicable.

  ## Skill Usage

  ### Use these skills:
  - `superpowers:using-superpowers` (always first)
  - `superpowers:dispatching-parallel-agents` (for independent work streams — USE AGGRESSIVELY)
  - `superpowers:subagent-driven-development` (for parallel plan steps — USE AGGRESSIVELY)
  - `superpowers:test-driven-development` (for every implementation step, including in subagents)
  - `superpowers:verification-before-completion` (before final commit)

  ### Do NOT use these skills:
  - `brainstorming` — we already have an optimized plan
  - `executing-plans` — this preamble replaces it
  - `using-git-worktrees` — single branch workflow
  - `writing-plans` — plan is already provided
  - `writing-skills` — not relevant
  - `receiving-code-review` / `requesting-code-review` — post-implementation concern
  - `systematic-debugging` — only if unexpected failures occur
  - `finishing-a-development-branch` — use `/finish` after implementation

  ## Completion

  After all plan steps are implemented:
  1. Run `superpowers:verification-before-completion`
  2. Confirm all tests pass
  3. Report a summary of all commits created with their messages
  4. Ask the user if they want to run `/finish` to push and create a PR
