CLAUDE.md Files (empfohlen für Projekt-Kontext)
Der Inhalt von CLAUDE.md wird als erste User-Message in der Konversation eingefügt, direkt nach dem System Prompt. GitHub Das ist der Standardweg für projektspezifische Instruktionen. Die Dateien sind hierarchisch aufgebaut:

~/.claude/CLAUDE.md — globale Präferenzen
CLAUDE.md im Projekt-Root — projektweite Regeln
CLAUDE.md in Unterverzeichnissen — spezifischerer Kontext, der bei Relevanz priorisiert wird

Du kannst mit # im Chat schnell Einträge hinzufügen, die dann automatisch in die passende Datei geschrieben werden.
--append-system-prompt (für Session-spezifisches Verhalten)
Das --append-system-prompt-Flag fügt deine Instruktionen an das bestehende Standardverhalten von Claude an, anstatt es komplett zu ersetzen. ClaudeLog Es funktioniert sowohl im Print- als auch im Interactive-Modus:
bash# Einmaliger Aufruf mit Fokus
claude -p "Review this code" --append-system-prompt "Focus on security vulnerabilities"

# Interaktive Session mit persistentem Fokus
claude --append-system-prompt "You are a senior backend engineer..."
--system-prompt (kompletter Ersatz)
Damit kannst du den gesamten System Prompt durch deinen eigenen ersetzen. Nützlich für sehr spezialisierte Einzelaufgaben, aber du verlierst dabei alle eingebauten Tool-Instruktionen.
Agent SDK (programmatisch)
Im Agent SDK kannst du den Claude Code Preset mit einer append-Property nutzen, um eigene Instruktionen hinzuzufügen und gleichzeitig die eingebaute Funktionalität zu erhalten. Claude Das sieht so aus:
typescriptsystemPrompt: { type: "preset", preset: "claude_code", append: "Deine Regeln hier" }
tweakcc (inoffiziell, tiefgreifend)
Das Tool tweakcc von Piebald-AI erlaubt es, die einzelnen Teile des System Prompts als Markdown-Dateien anzupassen GitHub — also auch die internen Tool-Beschreibungen, Sub-Agent-Prompts usw. Die Änderungen werden über npx tweakcc angewendet und unterstützen sowohl npm- als auch native Installationen. GitHub

Für deinen Use Case mit Skills und CLAUDE.md-basierten Workflows dürfte die Kombination aus hierarchischen CLAUDE.md-Dateien und --append-system-prompt am sinnvollsten sein. tweakcc ist spannend, wenn du die internen Mechanismen wie Tool-Beschreibungen oder Sub-Agent-Prompts modifizieren willst — aber natürlich mit dem Risiko, bei jedem Claude Code Update Konflikte zu bekommen.

markdown# Tool Usage Rules (RTK Optimization)

CRITICAL: For ALL file and directory operations, use Bash commands instead of native tools.
This is required because a CLI proxy (rtk) intercepts bash commands to compress output and save tokens.

## Mandatory Bash alternatives:

- **Reading files**: Use `cat <file>` or `head -n <N> <file>` — NEVER use the native Read tool
- **Listing directories**: Use `ls` or `find` — NEVER use the native ListDirectory tool  
- **Searching code**: Use `grep -r` or `rg` — NEVER use the native Search/Grep tool
- **Git operations**: Always use `git status`, `git diff`, `git log` etc. via Bash
- **File diffs**: Use `diff` via Bash

## Operations that SHOULD still use native tools:
- **Writing/editing files**: Use the native Write/Edit tools (rtk cannot optimize write operations)
- **TodoWrite**: Continue using natively

## Why this matters:
A transparent CLI proxy rewrites bash commands to compressed equivalents (e.g., `cat file.rs` → `rtk read file.rs`).
This reduces token consumption by 60-90%. Native tool calls bypass this optimization entirely.

markdown# Tool Selection Override: Bash-First Policy

A CLI proxy (rtk) intercepts all Bash commands to compress their output by 60-90%, 
saving significant tokens. For this to work, you MUST route operations through Bash 
instead of using native tools.

- Reading files: `cat`, `head`, `tail` — NOT the native Read/View tool
- Listing directories: `ls`, `find`, `tree` — NOT the native ListDirectory tool
- Searching code: `grep -rn`, `rg` — NOT the native Search/Grep tool

- Running tests: `cargo test`, `npm test`, `pytest` via Bash
- Checking builds/lints: `cargo build`, `eslint`, `tsc` via Bash

- Write/Edit (file modifications) — rtk cannot optimize writes
- TodoWrite — no bash equivalent needed
- Bash tool itself — obviously

Native tool calls bypass this entirely and waste tokens.
